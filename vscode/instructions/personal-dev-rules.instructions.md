# 個人 Copilot ルール（個人開発 / 内製ツール向け）

## 適用範囲
- このファイルは、常時有効な個人開発ルールを定義する。
- コーディング、レビュー、リファクタリング作業に適用する。

## 事実ベース回答
- API、ファイル、コマンド、パッケージ挙動を捏造しない。
- 不確かな場合は、不明点と確認が必要な点を明示する。
- 具体的な編集提案の前に、関連ファイルを読む。
- 事実と推測を明確に区別する。

## Git 運用

### コミットメッセージの言語と形式
- コミットメッセージは必ず日本語で書く。
- 件名は簡潔に保つ（目安 50 文字前後）。
- 推奨形式:

```text
[type] 概要

必要なら詳細説明
```

### 使用するコミット種別
- `[feat]` 新機能
- `[fix]` バグ修正
- `[docs]` ドキュメントのみ
- `[refactor]` 挙動変更なしのリファクタリング
- `[test]` テスト追加・更新
- `[chore]` 保守作業

### ブランチ戦略
- `main`: 本番投入可能な状態のみを置く
- `develop`: 次リリース向けの統合ブランチ
- `feature/<topic>`: `develop` から切り、`develop` に戻す
- `bugfix/<topic>`: `develop` から切り、`develop` に戻す
- `hotfix/<topic>`: `main` から切り、`main` と `develop` に戻す
- `release/<version>`: リリース準備用。完了後に `main` と `develop` に戻す

### Pull Request の最低ルール
- 何を変えたかだけでなく、なぜ変えるのかを書く。
- マージ前にテストを通す。
- PR のスコープは小さく、目的を絞る。

## コーディング品質
- 巧妙さより、単純で読みやすいコードを優先する。
- 関数は小さく、組み合わせやすく保つ。
- 不要な依存は追加しない。
- プロジェクトに formatter / linter 設定がある場合はそれに従う。

## テスト
- 新しい挙動にはテストを追加する。
- バグ修正には回帰テストを追加する。
- 編集後は関連テストを実行し、結果を報告する。

## セキュリティ基準
- 秘密情報をハードコードしない。
- 外部入力は検証・サニタイズする。
- 機微な値をログ出力しない。

## ドキュメント
- 挙動変更があれば README やインラインドキュメントを更新する。
- 自明でない判断はコード近傍に記録する。

## AIエージェント間協調（AGENTS.md）

複数のAIエージェント（Copilot、Claude Code など）が同一プロジェクトに関与する場合、それぞれが独立して判断・行動すると意図の衝突や重複作業が生じる。これを防ぐため、各プロジェクトルートに `AGENTS.md` という共有ログファイルを置き、エージェント間の「引き継ぎノート」として機能させる。このファイルはコードや設計の変更意図・結果・状態を時系列で記録し、どのエージェントも行動前に読み、行動後に書く義務を持つ。

`AGENTS.md` はプロジェクト固有の一時的な作業記録であり、リポジトリに push する必要はない。必ず `.gitignore` に追記すること。

### 行動ルール

アクションを起こす前に `AGENTS.md` が存在するか確認し、存在すればファイル末尾を読んで未解決の記録を把握してからアクションに移る。ソースコードやドキュメントに変更を加えた場合は、必ずその内容を `AGENTS.md` 末尾に追記する。記録を確認した際に、`Status` が `done` であり他エージェントの追加アクションが不要と判断できる古いエントリは削除してよい。ファイルの肥大化を防ぐためであり、有効な情報を消してはならない。

### エントリフォーマット

```markdown
## [YYYY-MM-DD] {エージェント名} — {変更の概要}

**Intent**
なぜこの変更が必要だったのか、背景と意図を文章で説明する。単なる「何をしたか」ではなく、「なぜそうしたか」を書く。
他エージェントが読んだときに判断の根拠が伝わるよう、十分な文脈を持たせること。

**Files**
- path/to/file.ts（変更内容の簡潔な説明）
- path/to/other.md（同上）

**Result**
実際に何が変わったか、テストや動作確認の結果はどうだったかを文章で書く。
意図通りにいかなかった点や、次のエージェントへの申し送り事項があればここに書く。

**Status**: pending | in_progress | done
```

`Status` の意味は次の通り。`pending` は他エージェントの確認や追加アクションを待っている状態。`in_progress` は自分が現在作業中であることを示し、他エージェントへの競合防止の宣言でもある。`done` はこのエントリに関する作業が完結しており、次回確認時に削除してよいことを示す。

## プロジェクト固有ルールとの優先順位
- プロジェクトに `.github/copilot-instructions.md` がある場合は、そのプロジェクト固有ルールでこの既定値を上書きできる。
