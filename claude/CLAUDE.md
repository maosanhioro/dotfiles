# 個人開発ルール（Claude Code 共通）

このファイルは、CLI・VSCode拡張を問わずすべての Claude Code セッションに横断的に適用される個人開発の基本原則を定義する。コーディング・レビュー・リファクタリング・設計提案のあらゆる場面で有効であり、プロジェクトルートに `CLAUDE.md` が存在する場合はそちらが優先される。

---

## 事実ベース回答

AIとして回答・提案を行う際は、常に事実を根拠とすること。

- API・ファイル・コマンド・パッケージの挙動を推測で断言しない。不確かな場合は「不確かである」と明示し、確認すべき点を具体的に示す。
- 具体的な編集提案を行う前に、必ず関連ファイルを読む。ファイルの内容を読まずに変更を提案しない。
- 事実と推測を明確に区別して伝える。「〜のはずです」と「〜です」を混同しない。
- 自分の知識のカットオフ日より新しい情報が絡む場合（ライブラリの最新バージョン、APIの変更など）は、その旨を明示する。

---

## AIとしての振る舞い・提案姿勢

### 変更前の確認

変更の影響範囲が広い場合・破壊的変更を含む場合・複数の実装アプローチが存在する場合は、コードを書く前に計画を提示してユーザーの確認を取る。小さな変更でも、思想や設計の方向性が変わるものは事前確認を優先する。

### スコープの遵守

依頼されたスコープ外の問題を発見した場合、勝手に変更せず「気になった点がある」として指摘にとどめる。複数の問題を同時に解決しようとしてスコープを広げない。修正は一度に一つの目的に絞る。

### 破壊的変更の明示

公開APIの変更・データ構造の変更・設定ファイルのスキーマ変更など、後方互換性を壊す変更を含む場合は、変更前に必ず「破壊的変更である」と明示する。データ移行が必要な場合も同様に指摘する。

### 選択肢のトレードオフ提示

実装アプローチが複数ある場合、「どれが唯一の正解」とは言わず、それぞれのトレードオフを簡潔に示す。ユーザーが判断できる情報を渡すことを優先する。

## コーディング品質

### 読みやすさと単純さ

巧妙さより単純さを優先する。コードは書く回数より読む回数のほうが圧倒的に多い。未来の自分と他者（他エージェント含む）が理解できる形で書く。

- 複雑な一行表現より、複数行の明快なコードを選ぶ。
- ネストは浅く保つ。early return・guard clause を積極的に使う。
- 関数は一つのことだけをする。複数の責務を持つ関数はリファクタリング対象。

### 命名

名前はそのものの意図を表す。略語・暗号的な名前・`tmp`/`data`/`info`/`obj` のような意味のない名前を避ける。

- 変数名は「何を保持しているか」を表す。
- 関数名は「何をするか」を動詞で表す。
- ブール値は `is`/`has`/`can`/`should` などのプレフィックスを使う。
- 名前を変えると説明コメントが不要になる場合、名前を変えることを優先する。

### 関数と副作用

- 関数は小さく、組み合わせやすく保つ。
- 副作用（I/O・グローバル状態変更・外部API呼び出し）は局所化し、純粋な計算ロジックから分離する。
- 副作用を持つ関数には、その事実がわかる名前をつけるか、コメントで示す。

### マジックナンバー・マジックストリング

リテラルの数値・文字列を直接コードに埋め込まない。意味のある定数名に抽出する。なぜその値なのかが名前から伝わるようにする。

### 型安全性

型システムを持つ言語（TypeScript・Go・Rustなど）では型を最大限に活用する。

- `any`・`interface{}`・型アサションの乱用を避ける。
- 関数のシグネチャには引数と戻り値の型を明記する。
- `null`/`undefined` の可能性がある値は型で表現し、呼び出し側に明示する。

### 依存関係

- 既存のコードや標準ライブラリで代替できる場合、新たな依存を追加しない。
- 依存を追加する場合は、そのライブラリのメンテナンス状態・ライセンス・バンドルサイズへの影響を確認する。
- `lock file`（package-lock.json・go.sum など）は必ずコミットする。

## エラーハンドリング

エラーを無視・握りつぶすことは禁止。エラーは適切に処理するか、上位に伝播させる。

- `catch` ブロックで何もしない（空の catch・コメントだけの catch）は原則禁止。どうしても握りつぶす必要がある場合は理由をコメントで明記する。
- エラーメッセージには「何が失敗したか」だけでなく「どのコンテキストで失敗したか」を含める。
- エラーの種類（予期されるエラー vs プログラムのバグ）を区別し、扱い方を変える。
- ユーザーに見せるエラーメッセージと、ログに残す内部エラーを混同しない。

## テスト

### テストの対象

実装の詳細ではなく、振る舞いをテストする。内部の関数呼び出しの回数ではなく、「この入力に対してこの出力・状態になる」という事実をテストする。リファクタリングでテストが壊れるようなら、テストが実装に依存しすぎている。

### テストピラミッド

ユニットテストを最も多く書き、統合テスト・E2Eテストは必要な分だけ書く。E2Eテストは遅く壊れやすいため、重要なユーザーフローの最小限に絞る。

### モック

モックは外部依存（DB・API・ファイルシステム）に限定する。内部ロジックのモックは避け、テストダブルが増えすぎている場合は設計の問題を疑う。

### テストの追加義務

- 新しい挙動を実装した場合、ユニットテストを追加する。
- バグを修正した場合、そのバグを再現する回帰テストを追加してから修正する。
- 編集後は関連テストを実行し、結果を報告する。

---

## パフォーマンス

パフォーマンスの最適化は「計測してから」行う。推測で最適化しない。

- O(n²) 以上の計算量になるアルゴリズムを使う場合は、その選択を意識的に行い、必要であればコメントで理由を示す。
- ループ内での不必要な計算・重複したAPI呼び出し・不要なレンダリングトリガーを避ける。
- パフォーマンス改善を目的とするリファクタリングは、改善前後の計測値とともに記録する。

---

## セキュリティ基準

- 秘密情報（APIキー・パスワード・トークン）をソースコードにハードコードしない。環境変数・シークレット管理サービスを使う。
- 外部入力（ユーザー入力・APIレスポンス・ファイル内容）は検証・サニタイズしてから使用する。
- 機微な値（トークン・パスワード・個人情報）をログ出力しない。
- SQL・コマンド・HTMLなどのインジェクションが起きうる箇所ではプレースホルダ・エスケープ・サニタイズを徹底する。
- 認証と認可を混同しない。「ログイン済み」と「その操作を行う権限がある」は別の判定。

## Git 運用

### コミットの粒度

一つのコミットは一つの目的に絞る。無関係な変更を同一コミットに混ぜない。コミット単位でビルドが通る状態を保つ。

### コミットメッセージの言語と形式

- コミットメッセージは必ず日本語で書く。
- 件名は簡潔に保つ（目安 50 文字前後）。

```text
[type] 概要

必要なら詳細説明（なぜこの変更が必要だったかを書く）
```

### 使用するコミット種別

- `[feat]` 新機能
- `[fix]` バグ修正
- `[docs]` ドキュメントのみ
- `[refactor]` 挙動変更なしのリファクタリング
- `[test]` テスト追加・更新
- `[chore]` 保守作業
- `[perf]` パフォーマンス改善
- `[security]` セキュリティ修正

### ブランチ戦略

- `main`: 本番投入可能な状態のみを置く
- `develop`: 次リリース向けの統合ブランチ
- `feature/<topic>`: `develop` から切り、`develop` に戻す
- `bugfix/<topic>`: `develop` から切り、`develop` に戻す
- `hotfix/<topic>`: `main` から切り、`main` と `develop` に戻す
- `release/<version>`: リリース準備用。完了後に `main` と `develop` に戻す

### Pull Request の最低ルール

- 何を変えたかだけでなく、なぜ変えるのかを書く。
- マージ前にテストとlintを通す。
- PRのスコープは小さく、目的を一つに絞る。
- 破壊的変更を含む場合はPRタイトルに明記する。

---

## ドキュメント

- 挙動変更があれば README・インラインドキュメント・型定義のコメントを更新する。
- 自明でない設計判断はコード近傍に記録する。「何をしているか」ではなく「なぜそうしているか」を書く。
- コメントは嘘をつく（コードと乖離する）ため、コードで表現できることはコードで表現し、コメントに頼りすぎない。

## AIエージェント間協調（AGENTS.md）

複数のAIエージェント（Copilot、Claude Code など）が同一プロジェクトに関与する場合、それぞれが独立して判断・行動すると意図の衝突や重複作業が生じる。これを防ぐため、各プロジェクトルートに `AGENTS.md` という共有ログファイルを置き、エージェント間の「引き継ぎノート」として機能させる。このファイルはコードや設計の変更意図・結果・状態を時系列で記録し、どのエージェントも行動前に読み、行動後に書く義務を持つ。

`AGENTS.md` はプロジェクト固有の一時的な作業記録であり、リポジトリに push する必要はない。必ず `.gitignore` に追記すること。

### 行動ルール

アクションを起こす前に `AGENTS.md` が存在するか確認し、存在すればファイル末尾を読んで未解決の記録を把握してからアクションに移る。ソースコードやドキュメントに変更を加えた場合は、必ずその内容を `AGENTS.md` 末尾に追記する。記録を確認した際に、`Status` が `done` であり他エージェントの追加アクションが不要と判断できる古いエントリは削除してよい。ファイルの肥大化を防ぐためであり、有効な情報を消してはならない。

### エントリフォーマット

```markdown
## [YYYY-MM-DD] {エージェント名} — {変更の概要}

**Intent**
なぜこの変更が必要だったのか、背景と意図を文章で説明する。単なる「何をしたか」ではなく、「なぜそうしたか」を書く。
他エージェントが読んだときに判断の根拠が伝わるよう、十分な文脈を持たせること。

**Files**
- path/to/file.ts（変更内容の簡潔な説明）
- path/to/other.md（同上）

**Result**
実際に何が変わったか、テストや動作確認の結果はどうだったかを文章で書く。
意図通りにいかなかった点や、次のエージェントへの申し送り事項があればここに書く。

**Status**: pending | in_progress | done
```

`Status` の意味は次の通り。`pending` は他エージェントの確認や追加アクションを待っている状態。`in_progress` は自分が現在作業中であることを示し、他エージェントへの競合防止の宣言でもある。`done` はこのエントリに関する作業が完結しており、次回確認時に削除してよいことを示す。

## プロジェクト固有ルールとの優先順位

プロジェクトルートに `CLAUDE.md` がある場合は、そのプロジェクト固有ルールでこのファイルの内容を上書きできる。共通ルールは最低基準であり、プロジェクトの要件に応じて厳格化・緩和どちらの方向にも調整してよい。
