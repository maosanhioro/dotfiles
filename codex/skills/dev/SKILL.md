# Dev

# 目的

個人開発における「開発のやり方」を統一し、品質・セキュリティ・作業効率を安定させる。

# 原則

- 個人開発でもセキュリティと可用性の観点を省略しない。
- 安定性重視の技術選定を軸に、トレンド適合性の根拠を明示する。
- 企画・仕様・設計の間で、用語と前提を一貫させる。

# スコープ

- このスキルは全セッションで常時有効な個人共通ルールとして機能する。
- プロジェクト固有の SKILL.md がある場合はそちらを優先する。
- 対象: コーディング・レビュー・リファクタリングタスク全般。

# Git運用

- 細かな運用ルールは `CONTRIBUTING.md` や PRテンプレに寄せる。
- ここでは「運用方針の優先順位」と「例外時の判断基準」を扱う。
- Git設定の統一は `git config` テンプレで担保する。
- 短命ブランチで作業し、`main`へ小さく頻繁に統合する（TBD寄り）。
- マージ前に必ず `git pull --rebase` で最新化する。
- コミットメッセージは常に日本語で記述する。

# コーディング規約

- スタイルの細部は各言語のLint/Formatterに委ねる。
- ルールはツールで自動化し、手作業の規約は最小限にする。
- 命名や設計の原則は、レビュー観点として維持する。

# セキュリティ/品質

- 秘密情報: リポジトリにコミットしない（監査はツールで自動化する）。
- 依存関係: 更新/脆弱性チェックは自動化ツールで担保する。
- 入力検証: フレームワーク標準や共通モジュールに寄せる。
- 可観測性: 監視ツール/ログ基盤にルールを落とし込む。

# 事実ベース回答

- API、ファイル、コマンド、パッケージの挙動を推測で答えない。
- 不確かな場合は不明と明示し、確認が必要な点を示す。
- 変更提案の前に関連ファイルを読む。
- 事実と推測を明確に区別する。

# Codex運用

- 指示: ゴール・制約・期待する出力形式を最初に確認する。
- 変更: 影響範囲と意図を簡潔に記載する。
- 成果物: 重要な決定や理由を短く残す。
- 禁止: 破壊的操作や秘匿情報の扱いは事前確認する。

# テスト方針

- 詳細は `docs/testing.md` または `README` に集約する。
- ここでは「最低限守るべき基準」のみ記述する。
- テスト実行結果はCLI標準出力で確認できれば十分とする。

# リリース/運用

- 詳細は CI/CD 設定や運用手順に落とし込む。
- ここでは「失敗時の原則」と「段階的リリース方針」だけ扱う。

# ドキュメント更新

- 詳細は PRテンプレや `docs/` の運用ルールに寄せる。
- ここでは「更新を忘れないための合図」だけ残す。
# 追記方針

- 過剰に詳細な手順やテンプレは references/ に分離する。
- 反復的な生成が必要なら scripts/ や assets/ を追加する。
