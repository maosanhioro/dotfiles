# 開発の前提

1〜2人規模・複数プロジェクト並行・コンテキストスイッチが頻繁という前提で開発する。

- **少人数・長期運用**: 書いた本人が数年後に一人で保守する前提でコードを書く
- **運用コストを設計に織り込む**: 「今動く」より「1年後も一人で問題なく動かせる」を優先
- **可読性が最優先**: コンテキストスイッチが多いため、後で読み返しやすい明快さが必須
- **依存は負債**: 外部依存・マネージドサービスの追加は運用コストの増加として扱う

---

## 提案・設計方針

- 複数アプローチがある場合、「この規模・体制での推奨」を一つ明示し、理由を添える
- 「将来的に拡張できます」という案には維持コストも必ず示す
- 段階的な提案（MVP → 次のステップ）を優先する
- ライブラリ・アーキテクチャ選定では「障害時に一人で対処できる複雑さか」を判断基準に加える
- 設計相談は抽象レベルで終わらず、データ構造・APIの形・運用手順まで具体化する

---

## 行動原則

- **変更前確認**: 影響範囲が広い・破壊的・アプローチが複数ある場合は、コードを書く前に計画を提示する
- **スコープ遵守**: 依頼スコープ外の問題は指摘にとどめ、勝手に変更しない
- **破壊的変更の明示**: API・データ構造・設定スキーマの後方互換を壊す変更は事前に明示する
- **ファイルを読んでから変更**: 推測で変更を提案しない。関連ファイルを必ず読んでから提案する
- **事実と推測を区別**: 不確かな情報は不確かと明示する

---

## コーディング原則

- 巧妙さより単純さ。ネストは浅く、early return を積極的に使う
- 関数は一つのことだけをする
- 名前はその意図を表す。`tmp`/`data`/`obj` のような意味のない名前を避ける
- 型システムを最大限に活用する。`any`・型アサションの乱用を避ける
- コメントは「なぜ」だけ書く。何をしているかはコードが語る
- 既存コード・標準ライブラリで代替できる場合、新たな依存を追加しない
- lock ファイル（package-lock.json 等）は必ずコミットする
- パフォーマンス最適化は計測してから。推測で最適化しない

---

## エラーハンドリング

- エラーは握りつぶさない。空の `catch` は原則禁止（やむを得ない場合は理由をコメントで明記）
- エラーメッセージには「何が」「どのコンテキストで」失敗したかを含める
- 予期されるエラーとプログラムのバグを区別して扱う

---

## テスト

- 実装の詳細ではなく振る舞いをテストする
- 新しい挙動を実装したらユニットテストを追加する
- バグ修正前にそのバグを再現する回帰テストを書く
- モックは外部依存（DB・API・ファイルシステム）に限定する

---

## セキュリティ

- 秘密情報をソースコードにハードコードしない
- 外部入力はすべて検証・サニタイズしてから使用する
- SQL・コマンド・HTML インジェクションが起きうる箇所ではプレースホルダ・エスケープを徹底する

---

## Git 運用

コミットメッセージは日本語。一コミット一目的。

```text
[type] 概要（50字前後）
```

種別: `[feat]` `[fix]` `[docs]` `[refactor]` `[test]` `[chore]` `[perf]` `[security]`

## エージェント間協調

プロジェクトルートに `AGENT_HANDOFF_LOG.md` があれば、アクション前に末尾を読み、変更後に追記する。
`AGENTS.md` はこの用途には使わない（Codex CLI の指示ファイル名と衝突するため）。

### 委譲する作業の切り分け方針

監督役（Claude Code）が実装エージェント（Codex CLI / GitHub Copilot CLI 等）に
作業を委譲するときは、「機械的な実行」と「判断力が要る作業」を分けて考える。

- **委譲してよい**: テスト・lint・ビルドの実行と結果確認、指示された仕様どおりの実装
- **監督役が自分で担う**: テストケースの設計（何を・どう検証すべきか）、
  再現テストの内容決定、アーキテクチャ上の意思決定
- 理由: 実装エージェントは監督役とモデル性能が異なることが多く、判断力に品質が
  依存する作業を丸投げすると、テストの網羅性・正しさがモデル差でそのまま劣化する
- 委譲した実装にテストが含まれる場合、監督役はその成果物をそのまま信用せず、
  コードレビュー（/code-review 等）でカバレッジと妥当性を確認してから完了とみなす

**この基準はユーザーに委譲を頼まれた時の振り分けではなく、監督役自身が常に
適用する判断基準である。** 「委譲してよい」範囲に該当する作業（テスト・lint・
ビルドの実行、指示された仕様どおりの実装など）は、ユーザーから「実装させて」
「codexにやらせて」のような明示的な依頼がなくても、監督役が自発的に
`delegate` スキルの利用を検討し、該当すれば委譲する。全部自分で抱え込んで
実行しない。判断に迷う場合（規模が小さい・対象ペインの状態が不明など）は、
委譲するかどうかをユーザーに一言確認してから進めてよい。

### tmux 経由の委譲タスク（`[FROM: Claude Code]` で始まる指示）

隣のペインで動く Claude Code（監督役）からの依頼を受けたときのルール:

- 指示された範囲だけを実施する
- git commit / push / 破壊的操作は依頼に明記がない限り行わない
- 完了時はシェルで `dev report done` を実行し、最終行に `DEV_DONE` を単独行で出力する
- 続行不能時は `dev report blocked '理由'` を実行し、`DEV_BLOCKED: 理由` を単独行で出力する
  （`dev report` がステータスバーの色とデスクトップ通知を更新し、文字列は監督側が
  `dev peek` で読む。コマンドが失敗しても文字列出力は省略しない——それが縮退時の経路）

## プロジェクト固有ルール

プロジェクトルートに各ツールの指示ファイル（`CLAUDE.md` / `AGENTS.md`）がある場合、
そちらが本ファイルより優先される。Codex CLI・GitHub Copilot CLI はいずれも
`AGENTS.md` を直接読むため、専用のリンクファイルは不要。
